Políticas de Seguridad Corporativa

Normativas de gestión de contraseñas y accesos (Versión 2.0)

⚠️ Nota de Laboratorio

Esta página simula la política de seguridad oficial de la organización. Tu objetivo como estudiante es demostrar cómo la aplicación web incumple flagrantemente estas políticas mediante la inspección técnica y la explotación (Auth Bypass).

// POL-01: Complejidad de Contraseñas

REQ-1.1 Las contraseñas deben tener una longitud mínima de 8 caracteres.
REQ-1.2 Deben incluir al menos una letra mayúscula, un número y un carácter especial.
INCUMPLIDO El sistema actual permite contraseñas vacías y cualquier carácter, ya que solo procesa el hash resultante en el lado del cliente.

// POL-02: Almacenamiento Criptográfico

REQ-2.1 Está estrictamente prohibido el uso de MD5 o SHA1 para almacenar contraseñas debido a colisiones conocidas.
REQ-2.2 Toda contraseña debe almacenarse usando una función KDF fuerte (ej. bcrypt, Argon2) junto con un Salt aleatorio único por usuario.
INCUMPLIDO La API almacena hashes MD5 crudos sin Salt. Esto permite ataques de Rainbow Tables.

// POL-03: Transmisión y Autenticación

REQ-3.1 Toda autenticación debe transmitirse a través de un canal seguro cifrado (TLS/HTTPS).
REQ-3.2 El servidor debe validar la credencial original (secreto compartido), no un derivado computable públicamente por el cliente.
INCUMPLIDO El servidor acepta un hash como credencial directa ("Auth Bypass via Client-Side Hashing"), permitiendo la suplantación de identidad sin conocer la contraseña.

// POL-04: Privacidad de la Información

REQ-4.1 Las bases de datos de credenciales deben mantener el principio de mínimo privilegio.
INCUMPLIDO El endpoint público /api/users expone el 100% de los identificadores y hashes, violando gravemente la confidencialidad.
