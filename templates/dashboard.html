// Conclusiones de Seguridad del Laboratorio

❌ MD5 para Contraseñas

MD5 fue diseñado para velocidad (checksums), no para seguridad. Genera 2^128 posibilidades pero con hardware moderno se pueden calcular 10 billones de hashes/segundo. Rainbow tables precomputadas cubren la mayoría de contraseñas comunes.

❌ Hashing en Cliente ("Auth Bypass")

Si el hash se genera en el cliente, el hash ES la contraseña. Técnicamente, esto se conoce como Auth Bypass via Client-Side Hashing. En el laboratorio lo llamamos análogamente Pass-the-Hash porque la explotación es idéntica: quien captura el hash, puede enviarlo directamente como credencial sin conocer el texto original (a diferencia del PtH puro en entornos Windows/NTLM, pero el concepto de reutilización es el mismo).

❌ Sin Salt

Sin salt, dos usuarios con la misma contraseña tienen el mismo hash. Esto permite ataques de correlación: si crackeas un hash, automáticamente crackeas todos los que tengan el mismo.

✅ Solución Correcta

Usar bcrypt, scrypt o Argon2 en el servidor con salt aleatorio por usuario. Enviar la contraseña en texto plano por HTTPS/TLS (el canal cifrado protege el tránsito).

✅ bcrypt en Práctica

import bcrypt hash = bcrypt.hashpw( password.encode(), bcrypt.gensalt(rounds=12) ) # Verifica automáticamente el salt embebido bcrypt.checkpw(password.encode(), hash)
